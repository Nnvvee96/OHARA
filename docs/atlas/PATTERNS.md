## Part 1: Kern-Prinzipien

P#1: SOUL First
  Vor jeder Entscheidung: passt das zu SOUL.md?
  Konflikt → technische Entscheidung überdenken.

P#2: Append-Only Denken
  Supersede statt Delete. Deprecate statt Remove.
  Fehler sind Lernmaterial, kein Versagen.

P#3: Utility Vector Pflicht
  Kein Atom ohne Nutzvektor. OHARA ist Schatzkammer, kein Archiv.
  Nutzen ist breit: direkt, relational, spekulativ, zukunftsoffen.

P#4: Wizard-Personas sind funktional
  Persona formt Extraktion. Schwache Persona = schwache Epistemik.
  Wizards wachsen über Zeit — das ist das Design.

P#5: Skeptic Gate ist heilig
  EMERGING → VALIDATED ohne Skeptic = epistemische Korruption.
  Keine Ausnahmen, keine Bypässe.

P#6: Keine Duplikate
  Update statt Neuerstellen. Immer prüfen was existiert.

P#7: Secrets in .env only
  Niemals in Code, Chat oder Git.

P#8: Ketten erkennen
  Relationale Kausalität über mehrere Stufen.
  Einzelglieder können unscheinbar sein — die Kette ist wertvoll.

P#9: Drei DBs, nicht eine
  vault / knowledge / governance getrennt.

P#10: Wizard Evolution ist Design
  Fehler erinnern, Erfolge belohnen, Beförderung durch Verdienst.

P#11: Organisches Wachstum
  Roster, Hierarchien, Domänen entstehen durch Verdienst.
  Nie vorab over-engineered.

P#12: Reproduktion durch Verdienst
  Seniors bekommen Mitarbeiter wenn sie es verdient haben.
  Neue Domänen entstehen organisch.

P#13: OHARA übersteigt seinen Schöpfer
  Das ist das Ziel, kein Risiko.

P#14: Räume sind Ressourcen
  Regeneration ist Teil der Zivilisation.

P#15: World-2-Kreislauf
  Builder-Ergebnisse fließen als Wissen zurück in World 1.

P#16: Zusammenspiel übersteigt Einzelteile
  Cross-Domain-Resonanz ist das wertvollste Wissen.
  Wissende die gemeinsam arbeiten produzieren was keiner alleine kann.

P#17: Beispiele sind Illustrationen
  OHARA ist breiter als jedes Beispiel.
  Vom Prinzip denken — nicht vom Beispiel.

P#18: UI ist eine Welt, nicht ein Dashboard
  Begehbare Library. Räume. Wissende beobachtbar.
  Generiert kein Wissen — spiegelt es.
